# celestial-bodies-database
Celestial Bodies DatabaseA relational database representing a hierarchical model of our universe, built as part of the freeCodeCamp Relational Database Certification.🌌 Project OverviewThe goal of this project was to design and implement a persistent database using PostgreSQL that accurately models the relationships between various celestial bodies. The project emphasizes Schema Design, Normalization, and Referential Integrity.🛠️ Tech StackDatabase: PostgreSQLEnvironment: Docker / Linux TerminalLanguage: SQL📐 Database SchemaThe database follows a strict hierarchical "chain" structure to ensure data integrity and minimize redundancy.1. Entity Relationship SummaryGalaxy: The top-level container.Star: Each star belongs to one galaxy (1:N).Planet: Each planet orbits one star (1:N).Moon: Each moon orbits one planet (1:N).Comet: A standalone table for tracking non-orbital celestial objects.2. Table StructuresTablePrimary KeyForeign KeyKey ColumnsGalaxygalaxy_idN/Aname, age_in_millions_of_years, has_lifeStarstar_idgalaxy_idname, is_spherical, distance_from_earthPlanetplanet_idstar_idname, planet_type, has_lifeMoonmoon_idplanet_idname, is_spherical, descriptionCometcomet_idgalaxy_idname, comet_type, distance_from_earth🚀 Key Features & ConstraintsNormalization: Data is organized to prevent "orphaned" records (e.g., a moon cannot exist without a parent planet).Constraints:UNIQUE constraints on all names to prevent duplicate entries.NOT NULL constraints on essential identification fields.SERIAL data types for automated Primary Key generation.Data Diversity: Includes various SQL data types including INT, NUMERIC, BOOLEAN, VARCHAR(20), and TEXT.📖 How to RunEnsure PostgreSQL is installed or run via Docker.Log into the psql interactive terminal:Bashpsql --username=freecodecamp --dbname=postgres

🧠 Personal ReflectionsAs an aspiring Analytics Engineer, I focused heavily on the ERD (Entity Relationship Diagram) phase of this project. Deciding between a "Hub-and-Spoke" model versus a "Hierarchical Chain" allowed me to weigh the trade-offs between query complexity and data integrity. I opted for the hierarchical approach to ensure the highest level of relational accuracy.
